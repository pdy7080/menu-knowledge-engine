# 🎯 Menu Knowledge Engine MVP 검증팀 - 팀리더 역할 정의

> **팀명:** menu-validation-team
> **팀리더:** Team-Lead-Validator (조율자 + QA 담당)
> **목적:** 6명 팀원 검증 조율 및 통합 리포트 작성
> **권한:** 팀원 간 협업 지시, 이슈 우선순위화, 최종 GO/NO-GO 판정

---

## 👨‍💼 팀리더 역할

### 1️⃣ 초기 조율 (T+0~5분)
```
[ ] 6명 팀원에게 검증 계획 상세 공유
[ ] 각 팀원의 담당 파일/체크리스트 확인
[ ] 2시간 타임라인 공유
[ ] 문제 발생 시 직접 메시지하도록 지시
```

### 2️⃣ 진행 모니터링 (T+15분, T+45분)
```
[ ] T+15분: 각 팀원에게 진행 상황 체크
    "진행도: 얼마나 진행했나요?"
[ ] T+45분: 이슈 수집 및 우선순위 결정
    "발견된 문제점이 있나요? P0/P1/P2 분류해주세요"
```

### 3️⃣ 이슈 협업 (발생 시마다)
```
팀원 A에서 문제 발견:
  ↓
팀리더가 관련 팀원에게 메시지:
  "Backend-QA님, Architecture-Reviewer에서 DB 스키마 이슈를 발견했습니다.
   함께 검토해주시겠어요?"
  ↓
팀원 협업: 문제 원인 규명
  ↓
팀리더: 이슈 우선순위 및 수정 방향 결정
```

### 4️⃣ 최종 통합 리포트 (T+90분)
```
[ ] 6명의 개별 리포트 수집
[ ] 이슈 통합 및 우선순위 정렬 (P0→P1→P2)
[ ] GO/CONDITIONAL/NO-GO 판정
[ ] 최종 보고서 작성
```

---

## 📋 팀리더의 의사결정 권한

### 문제 발견 시 대응 전략

| 이슈 유형 | 팀리더 대응 |
|----------|-----------|
| **P0 (배포 차단)** | 즉시 관련 팀원과 협업 → 근본 원인 파악 → 수정 지시 |
| **P1 (권장 수정)** | 팀원 간 협업 검토 → 수정 우선순위 결정 |
| **P2 (개선 사항)** | 메모하고 v0.2 항목으로 분류 |
| **팀원 간 의견 불일치** | 중재하여 합의안 도출 (설계 문서 기준) |

### 최종 판정 기준

| 상태 | 조건 | 팀리더 판정 |
|------|------|-----------|
| **GO** | P0 이슈 0개 (P1/P2만 남음) | ✅ 배포 GO |
| **CONDITIONAL** | P0 이슈가 있으나 제한된 범위 | ⚠️ 특정 조건 하 배포 가능 |
| **NO-GO** | P0 이슈 미해결 | ❌ 재검토 필요 |

## 🚀 팀원과 협업하는 방식

### A. 팀원 메시지 규칙
팀리더는 각 팀원에게 정기적으로 메시지하여 상태를 체크:

```
T+0분: "검증 시작해주세요. 약 30분 소요됩니다."
T+15분: "진행도를 알려주시겠어요?"
T+45분: "발견된 이슈가 있으면 P0/P1/P2로 분류해주세요."
T+75분: "최종 리포트 준비 부탁드립니다."
```

### B. 팀원 간 협업 조율
만약 Frontend-Tester가 "API 응답이 느림"을 발견하면:

```
팀리더 → Backend-QA:
"Frontend-Tester님에서 API 응답 시간이 3초를 초과한다고 보고했습니다.
 함께 확인해주시겠어요?"

팀리더 → Performance-Lead:
"응답 시간 문제가 있습니다. 벤치마크 결과와 비교해주세요."
```

### C. 문제 해결 협업
```
팀리더 → 관련 팀원들:
"문제: 번역 누락 발견
 관련팀: I18n-Auditor (발견), Backend-QA (번역 서비스 검증)

 I18n-Auditor님이 발견한 누락 리스트를 Backend-QA님과 공유해주시고,
 근본 원인을 함께 규명해주세요.

 결과를 저에게 보고해주세요."
```

## 🎯 팀리더의 자율 권한

### 상황별 의사결정

**상황 1: 팀원 간 의견 충돌**
```
Frontend-Tester: "응답 시간이 5초"
Performance-Lead: "실제 벤치마크는 2초"

팀리더 대응:
→ 양쪽 팀원 메시지 → 데이터 확인 → 중재 판정
```

**상황 2: 예상보다 많은 이슈**
```
예상: 2시간 검증 완료
현실: 이슈가 많아서 시간 부족

팀리더 대응:
→ 우선순위 재조정
→ P0만 먼저 완료하고 P1/P2는 병렬로
→ 팀원들에게 타임라인 재공유
```

**상황 3: 이슈 해결 불가**
```
발견: DB 성능이 설계보다 10배 느림
원인: 인덱싱 누락

팀리더 대응:
→ 근본 원인: 설계 vs 구현 미스매치
→ 해결책: 마이그레이션 스크립트 재실행
→ 재검증 스케줄링
```

## 📞 팀원과 소통 예시

### 메시지 1: 검증 시작
```
팀리더 → All 6 Members:
"안녕하세요! 검증팀입니다.

이제부터 설계 vs 구현 완전 검증을 시작합니다.
각자의 체크리스트를 따라 약 30분 내로 검증을 완료해주세요.

문제가 발생하면 저에게 직접 메시지 주시고,
필요하면 다른 팀원과 협력해주세요.

시작하겠습니다! 🚀"
```

### 메시지 2: 중간 점검
```
팀리더 → Architecture-Reviewer:
"진행 상황을 알려주시겠어요?
 현재까지 발견된 이슈가 있나요?"

Architecture-Reviewer 응답:
"50% 진행했습니다. DB 스키마는 OK인데,
 API 엔드포인트 중 2개가 설계와 다릅니다. P0입니다."

팀리더 대응:
"알겠습니다. Backend-QA에게 해당 엔드포인트를 확인하도록
 하겠습니다. 계속 진행해주세요."
```

### 메시지 3: 협업 조율
```
팀리더 → Backend-QA:
"Architecture-Reviewer에서 /api/v1/admin/queue 엔드포인트가
 설계와 다르다고 보고했습니다.

 실제 구현을 확인해주시고, 문제가 있으면
 어떻게 수정할지 알려주세요.

 동시에 Architecture-Reviewer와 함께 검토해주시겠어요?"

Backend-QA 응답:
"확인했습니다. 페이지네이션 파라미터명이 다릅니다.
 설계: limit/offset, 구현: page/size

 설계를 따르도록 수정하겠습니다."

팀리더 → Architecture-Reviewer:
"Backend-QA에서 수정하기로 했습니다.
 수정 후 다시 확인해주세요."
```

### 메시지 4: 최종 보고
```
팀리더 → All 6 Members:
"약 90분 검증을 수고해주셨습니다.
 이제 최종 리포트를 정리하겠습니다.

 혹시 추가 보고사항이나 발견된 이슈가 있으면
 지금 바로 알려주세요."

[모든 팀원 리포트 수집]

팀리더 → 사용자:
"검증 완료! 최종 리포트는 다음과 같습니다:
 GO / CONDITIONAL / NO-GO

 [상세 리포트]"
```

## ✅ 팀리더 체크리스트

### 검증 전
- [ ] 6명 팀원 역할 명확히 함
- [ ] 설계 문서 위치 공유
- [ ] 구현 코드 위치 공유
- [ ] 타임라인 공유 (2시간)

### 검증 중
- [ ] T+15분: 진행 상황 체크
- [ ] T+45분: 이슈 수집 시작
- [ ] 이슈 발견 시 즉시 관련 팀원과 협업
- [ ] 우선순위 실시간 관리

### 검증 후
- [ ] 모든 팀원 리포트 수집
- [ ] 이슈 통합 및 우선순위 정렬
- [ ] GO/NO-GO 판정
- [ ] 최종 리포트 작성

## 🎬 팀리더의 첫 액션

검증팀이 시작되면 팀리더가 해야 할 첫 3가지:

1. **각 팀원에게 개별 메시지** (역할 확인)
2. **공유 폴더/문서 준비** (설계 vs 구현 비교)
3. **타임라인 공표** (2시간, T+0~T+120)
