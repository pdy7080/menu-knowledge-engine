# 03. 양방향 데이터 전략

## 1. 핵심 인사이트

> **데이터 수집을 B2B(식당 등록)에만 의존하면 성장이 느리다.  
> B2C(외국인 촬영)에서 역방향으로도 데이터가 흘러들어와야  
> 양면시장 플라이휠이 완성된다.**

---

## 2. 정방향 흐름 (B2B → 외국인)

### 기본 흐름

```
식당 사장님이 메뉴 사진 업로드
 → OCR (텍스트 추출)
 → 메뉴명 후보 리스트 생성
 → Menu Knowledge Engine 호출
   ├── DB에 이미 있는 메뉴 → 기존 Canonical 연결 (즉시)
   └── DB에 없는 메뉴 → AI + 웹 서치 탐색 → 신규 등록
 → 사장님 확인/수정
 → 다국어 QR 메뉴 페이지 자동 생성
 → 외국인 관광객이 QR 스캔하여 열람
```

### 사장님이 보는 화면

```
┌─────────────────────────────────┐
│ ✅ 메뉴 인식 완료               │
│                                 │
│ 할머니뼈해장국                    │
│ → 표준 메뉴: 뼈해장국 계열        │
│ → 외국인 표시명: Traditional      │
│   Pork Bone Soup                │
│ → 추천 이미지 있음               │
│ → 설명 자동 생성됨               │
│                                 │
│ [확인] [수정] [내 사진으로 교체]    │
└─────────────────────────────────┘
```

**핵심 UX 원칙:**
- 사장님은 "AI"를 의식하지 않음
- "자동으로 잘 만들어졌네" 정도의 경험
- 메뉴 사진만 올리면 끝 → 진입 장벽 최소화

## 3. 🔥 역방향 흐름 (B2C → DB 확장)

### 이것이 "게임 체인저"인 이유

```
외국인이 식당에서 메뉴판 촬영 (웹앱, 앱 설치 불필요)
 → OCR
 → Menu Knowledge Engine DB 매칭
   ├── 매칭 성공 → 즉시 설명 제공 (AI 호출 ❌)
   ├── 부분 매칭 → 수식어 분해로 추론 → 설명 제공
   └── 매칭 실패 → AI 탐색 → 결과 표시 + DB 자동 추가
 → 하단에: "이 식당의 전체 메뉴를 보시겠어요?
           사장님께 추천해주세요 → [Share link]"
```

### B2C 역방향이 해결하는 3가지 문제

**① B2C 경험이 끊기지 않음**
- "이 식당은 미등록이라 안 돼요" ❌
- DB에 개념만 있으면 어디서든 작동
- 외국인 입장: '한국 음식 설명 앱'처럼 느껴짐

**② 엔진이 스스로 성장함**
- 사용자가 새 메뉴를 찍을 때마다 미등록 메뉴 발견
- AI 탐색 → 신규 개념 생성 → DB 자동 확장
- **사용자 = 무료 데이터 수집자**

**③ 영업이 '권유'가 아니라 '추천'이 됨**
- "이 식당은 아직 등록 안 됐어요. 사장님께 추천해주세요"
- 광고 ❌ / 영업 ❌ / 사용자 추천 기반 네트워크 효과 ⭕

## 4. 데이터 플라이휠 (양면시장 선순환)

```
       ┌─── 외국인 사용자 증가 ◄──────┐
       │                              │
       ▼                              │
  메뉴 스캔 증가                   서비스 품질 향상
       │                              │
       ▼                              │
  미등록 메뉴 발견 → DB 성장 ──►  더 많은 메뉴 커버
       │                              │
       ▼                              │
  "이 식당 추천" 공유 ──────►   식당 가입 증가
       │                              │
       ▼                              │
  등록 식당의 전체 메뉴 DB화 ──►  지역 커버리지 확대
       │                              │
       └──────────────────────────────┘
```

**핵심:** B2B만으로는 이 루프가 안 돌아감. B2C 역방향이 있어야 플라이휠이 완성됨.

## 5. 숨겨진 데이터 자산

역방향 흐름에서 자동으로 쌓이는 데이터:

| 데이터 | 가치 |
|---|---|
| 외국인이 가장 많이 촬영한 메뉴 | → "이해 어려운 메뉴 TOP 50" 콘텐츠 |
| 지역별 스캔 빈도 | → "외국인 밀집 지역" 실시간 파악 |
| 언어별 조회 차이 | → 국가별 한식 관심도 분석 |
| 미등록 식당 위치 | → 신규 영업 타겟 리스트 |
| 시간대별 사용 패턴 | → 관광 행동 데이터 |

**이건 번역 서비스가 절대 못 갖는 행동 데이터(behavioral data)**

### 데이터 판매 가능 대상:
- 지자체/관광공사: "우리 지역 외국인 메뉴 이해도 리포트"
- 외식업 컨설팅: "외국인 선호 메뉴 트렌드"
- 배달앱: "외국인 사용자 확대를 위한 메뉴 인사이트"

## 6. 데이터 수집 이중 트랙 (편향 방지)

### 문제: 성수야만으로 시작하면 DB가 편향됨

- 성수동 = 퓨전/카페/트렌디 메뉴 위주
- 외국인이 진짜 헤매는 메뉴 = 곰탕, 순대국, 족발, 백반 등 로컬 한식

### 해결: Phase 1부터 이중 트랙

```
[트랙 A] 성수야 — 실사용 + 자동 수집
  - 성수야에 등록되는 가게 메뉴를 자동으로 엔진에 투입
  - 실제 사용자 피드백 수집 가능
  - 자연스럽게 쌓이는 데이터

[트랙 B] 관광 핵심지 — 의도적 현장 수집 (편향 제거)
  - 명동 / 종로 / 남대문 / 동대문
  - "외국인이 실제로 메뉴판 앞에서 멈칫하는 식당" 위주
  - 선정 기준:
    · 한글 메뉴만 있는 곳
    · 로컬 한식 (체인 ❌, 노포 ⭕)
    · 관광객 밀집 지역
  - 현장 촬영 기반 (크롤링 ❌, 직접 사진 ⭕)
```

> **성수야는 실험장, 명동/종로는 엔진의 근육**
