# 01. Concept Overview — Menu Knowledge Engine이란 무엇인가

## 한 줄 정의

> **한국 음식 메뉴를 "번역 대상"이 아닌 "정의된 지식 객체"로 전환하여,  
> 다국어 설명·문화적 맥락·식이 정보를 구조화된 API로 제공하는 지식 엔진**

---

## 핵심 원칙 3가지

### 원칙 1: AI는 도구이지 엔진이 아니다

AI(LLM)는 **초기에 메뉴의 정체성을 탐색·확정하는 단계**에서만 집중 사용한다.
확정된 지식은 DB에 저장되며, 이후 동일 메뉴에 대해서는 AI를 호출하지 않는다.

```
AI의 역할: "이 메뉴가 무엇인지" 밝혀내는 조사관
DB의 역할: "밝혀진 지식"을 저장하고 서빙하는 도서관
```

### 원칙 2: 메뉴는 문자열이 아니라 개념이다

"할머니뼈해장국"은 문자열이 아니라 아래와 같은 개념의 조합이다:
- **뼈해장국** (표준 메뉴: 돼지등뼈 해장국)
- **할머니** (수식어: 가정식 감성)

이 "개념 단위"로 저장하면 처음 보는 메뉴명도 분해·추론이 가능하다.

### 원칙 3: 양방향으로 데이터가 흘러야 한다

```
B2B (식당 → 엔진): 사장님이 메뉴 등록 → DB 성장
B2C (외국인 → 엔진): 관광객이 메뉴판 촬영 → 미등록 메뉴 발견 → DB 성장
```

한쪽만으로는 플라이휠이 안 돈다. 양방향이어야 자기강화 루프가 완성된다.

---

## 엔진의 3단계 작동

### Stage 1: Identity Discovery (AI 집중)
- 새로운 메뉴명을 만나면 AI + 웹 서치로 정체를 조사
- 여러 출처를 교차 검증하여 확신도(confidence)를 산정
- 결과를 구조화된 "메뉴 개념 객체"로 변환

### Stage 2: Canonicalization (정규화)
- 유사/동일 메뉴 간 관계를 정의 (variant, similar, confused_with 등)
- 수식어를 분리하여 Modifier Dictionary에 등록
- 같은 메뉴가 다른 식당에 등장해도 AI 재호출 불필요

### Stage 3: Knowledge Serving (DB 서빙)
- 모든 조회는 DB에서 처리 (AI 호출 없음)
- 수식어 자동 분해로 미등록 변형도 즉시 대응
- 다국어 번역·문화적 설명·식이 정보를 API로 제공

---

## 왜 이것이 "번역 서비스"가 아닌가

| 관점 | 번역 서비스 | Menu Knowledge Engine |
|---|---|---|
| 입력 | 텍스트 | 메뉴 "개념" |
| 처리 | 매번 AI 번역 | 1회 학습 → DB 서빙 |
| 출력 | 번역된 텍스트 | 구조화된 지식 객체 |
| 비용 곡선 | 사용량 비례 증가 | 시간에 따라 하락 |
| 자산 | 없음 | Knowledge Graph |
| 확장성 | 없음 | API / SaaS / 라이선싱 |
